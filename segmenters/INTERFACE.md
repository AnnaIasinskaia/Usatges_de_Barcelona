# Контракт сегментеров

## Назначение

Документ описывает унифицированный интерфейс (контракт) для всех модулей сегментации, расположенных в папке `segmenters/`. Цель контракта — обеспечить совместимость сегментеров с основным пайплайном обработки исторических текстов (`pipeline.py`, `pipeline_gramoty.py`) и упростить интеграцию новых источников.

Контракт определяет:
- сигнатуру функции сегментации;
- ожидаемые параметры;
- формат возвращаемых данных;
- требования к валидации и обработке ошибок.

## Унифицированный интерфейс

Все сегментеры должны реализовывать функцию с именем `segment_<source>_unified` (где `<source>` — условное имя источника), соответствующую следующей сигнатуре:

```python
from typing import List, Tuple, Union
from pathlib import Path

def segment_source_unified(
    source_file: Union[str, Path],
    source_name: str,
    min_words: int = 10,
    max_words: int = 150
) -> List[Tuple[str, str]]:
    """
    Разбивает текст исторического документа на смысловые сегменты.

    Параметры
    ---------
    source_file : str или Path
        Путь к файлу с исходным текстом (формат .txt, .docx или иной).
        Сегментер обязан прочитать файл самостоятельно.
    source_name : str
        Имя источника (например, "CorpusJuris", "Usatges", "Gramoty911").
        Используется для формирования идентификаторов сегментов и выбора
        специализированной логики сегментации.
    min_words : int, optional (по умолчанию 10)
        Минимальное количество слов в сегменте. Сегменты короче этого предела
        должны быть отброшены или объединены с соседними.
    max_words : int, optional (по умолчанию 150)
        Максимальное количество слов в сегменте. Если сегмент превышает этот
        предел, он должен быть разбит на части.

    Возвращает
    ----------
    List[Tuple[str, str]]
        Список сегментов, где каждый элемент — кортеж из двух строк:
        (segment_id, segment_text).

        - segment_id : уникальный идентификатор сегмента в пределах источника.
          Рекомендуемый формат: `{source_name}_{тип}{номер}`,
          например `Usatges_Art1`, `CorpusJuris_Ch2_P3`.
        - segment_text : очищенный текст сегмента (без разметки, номеров страниц,
          аппаратных помет). Должен содержать не менее 20 символов.

    Исключения
    ----------
    - FileNotFoundError: если `source_file` не существует.
    - ValueError: если параметры противоречивы (например, `min_words > max_words`).
    - TypeError: если возвращаемые данные не соответствуют ожидаемому формату.
    """

    # Реализация сегментера
    ...
```

## Детализация параметров

### source_file
Основной способ указать исходный документ. Сегментер должен уметь читать файлы форматов:
- `.txt` — plain text в UTF‑8;
- `.docx` — документы Microsoft Word (используется библиотека `python-docx`).

Сегментер не должен ожидать передачи текста через другие параметры; весь контент извлекается из файла. Если вызывающая сторона имеет текст в памяти, она должна сохранить его во временный файл перед вызовом (либо использовать адаптер).

### source_name
Строковый идентификатор источника, соответствующий ключу в `SOURCE_CONFIGS` (см. `config.py`). Используется для:
- выбора правил сегментации (например, разные регулярные выражения для разных источников);
- генерации `segment_id`;
- логирования и отладки.

### min_words и max_words
Ограничения на длину сегментов в словах (подсчёт слов — через `str.split()`). Сегментер должен гарантировать, что каждый возвращаемый сегмент содержит не менее `min_words` и не более `max_words` слов. Если исходный фрагмент (например, статья) превышает `max_words`, он разбивается на части с сохранением нумерации (например, `Art1_P1`, `Art1_P2`). Если фрагмент короче `min_words`, он может быть объединён с предыдущим/следующим фрагментом либо отброшен (в зависимости от логики источника).

## Возвращаемое значение

### Формат
Список кортежей фиксированной длины 2:
```python
[
    ("Usatges_Art1", "Text of article 1..."),
    ("Usatges_Art2", "Text of article 2..."),
    ...
]
```

### Требования к segment_id
- Уникален в пределах одного вызова сегментера.
- Содержит имя источника (или его аббревиатуру) для удобства отладки.
- Желательно сохранять исходную нумерацию документа (номера глав, статей, рубрик).
- Не должен содержать пробелов и специальных символов, кроме подчёркивания.

### Требования к segment_text
- Очищен от аппаратных помет, номеров страниц, колонтитулов, сносок.
- Нормализованы пробелы (нет множественных пробелов, неразрывных пробелов).
- Восстановлены переносы слов (слиты части слова, разделённые дефисом и переводом строки).
- Не содержит маркеров разметки (например, `[12]`, `*A`, `||`).
- Длина не менее 20 символов (проверяется функцией `validate_segments`).

## Валидация

Все сегментеры должны использовать функцию `validate_segments` из `seg_common.py` перед возвратом данных. Эта функция проверяет:
- тип возвращаемого значения (список кортежей);
- длину каждого кортежа (должна быть 2);
- типы элементов (`str`, `str`);
- минимальную длину текста (20 символов).

Если сегментер возвращает данные в другом формате (например, словари или кортежи из трёх элементов), необходимо преобразовать их к требуемому виду **до** вызова `validate_segments`.

## Пример реализации

```python
# segmenters/seg_example.py
from pathlib import Path
from typing import List, Tuple
import docx
import re
from .seg_common import validate_segments, clean_text

def segment_example_unified(
    source_file: str,
    source_name: str,
    min_words: int = 10,
    max_words: int = 200
) -> List[Tuple[str, str]]:
    # Извлечь текст из файла
    path = Path(source_file)
    if path.suffix == ".docx":
        doc = docx.Document(str(path))
        text = "\n".join(p.text for p in doc.paragraphs)
    else:
        text = path.read_text(encoding="utf-8")

    # Специфичная логика сегментации
    raw_segments = []
    for match in re.finditer(r"Article (\d+): (.+?)(?=Article|$)", text, re.DOTALL):
        art_num = match.group(1)
        art_text = clean_text(match.group(2))
        raw_segments.append((f"{source_name}_Art{art_num}", art_text))

    # Применить ограничения по длине
    segments = []
    for seg_id, txt in raw_segments:
        words = txt.split()
        if len(words) < min_words:
            continue
        if len(words) <= max_words:
            segments.append((seg_id, txt))
        else:
            # Разбить на части
            for i, start in enumerate(range(0, len(words), max_words), 1):
                chunk = " ".join(words[start:start + max_words])
                segments.append((f"{seg_id}_P{i}", chunk))

    # Валидация и возврат
    return validate_segments(segments, source_name)
```

## Стандартный блок main

Каждый сегментер должен содержать блок `if __name__ == "__main__"` для локального тестирования. Этот блок должен быть унифицирован и выполнять следующие действия:

1. Загрузить тестовый файл из папки `data/`, соответствующего источнику (например, `data/Evangelium.docx`).
2. Вызвать функцию сегментации с параметрами по умолчанию.
3. Вывести общее количество сегментов.
4. Вывести первые пять сегментов в формате `id: текст` (обрезанный до 100 символов).

Пример унифицированного блока:

```python
if __name__ == "__main__":
    from pathlib import Path
    import sys

    # Определить путь к тестовому файлу на основе source_name
    # (в реальном сегментере это может быть жёстко закодировано)
    test_file = Path("data/Example.docx")
    if not test_file.exists():
        print(f"Тестовый файл не найден: {test_file}", file=sys.stderr)
        sys.exit(1)

    # Вызвать сегментер 
    segments = segment_example_unified(test_file, "Example")

    # Вывод статистики
    print(f"Всего сегментов: {len(segments)}")
    print("\nПервые пять сегментов:")
    for i, (seg_id, seg_text) in enumerate(segments[:5], 1):
        preview = seg_text[:100] + ("..." if len(seg_text) > 100 else "")
        print(f"{i}. {seg_id}: {preview}")
```

**Примечание:** В текущей реализации многие сегментеры ожидают параметр `text`, а не `source_file`. В процессе миграции блок main должен быть адаптирован к новому интерфейсу (т.е. передавать `source_file` напрямую). До завершения миграции допускается использование адаптера, как показано выше.

## Примечания для разработчиков

- Новые сегментеры должны строго следовать описанному контракту.
- При изменении существующих сегментеров необходимо убедиться, что они остаются совместимыми с текущим пайплайном (через адаптеры в `source_segmenters.py`).
- Все сегментеры должны содержать блок `if __name__ == "__main__"` для локального тестирования на данных из папки `data/`.
- Дополнительные параметры, специфичные для источника, **не допускаются**. Вся необходимая конфигурация должна передаваться через `source_name` или извлекаться из структуры файла.
